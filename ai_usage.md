# AI Usage Report

This document describes how AI tools were used during Phase 1 of the project. 

## Tools Used

- **Cursor** (AI coding assistant in IDE):
  - Used to review code structure, suggest implementation steps, and identify missing deliverables.
  - Used for quick checks of build status and feature coverage.
- **Gemini**:
  - Used as a secondary assistant for idea validation and alternative implementation suggestions.
  - Used to compare approaches for permission handling and command argument design.

## Prompts I Gave

Examples of the kinds of prompts used:

- "Help me design a CLI structure for a C program with commands like add, list, view, remove, update threshold, and filter."
- "How should I safely create files/directories in C and enforce exact permission bits (0750, 0664, 0640, 0644)?"
- "Suggest a clean way to parse multiple filter conditions like `field:op:value` in C."
- "Review this function and point out edge cases for file access and permission checks."
- "What are common mistakes when using `stat`, `chmod`, `open`, and symlinks in Linux C programs?"

## What Was Generated by AI

AI-generated output included:

- Draft logic for command parsing and command dispatch.
- Suggestions for organizing code into modules (`main`, `operations`, `permissions`, `filter`).
- Example snippets for:
  - parsing conditions with `sscanf`,
  - validating access with POSIX permission bits,
  - creating/updating symbolic links for `active_reports-*`,
  - basic error handling with `perror`.
- Suggested test ideas for role-based access control and filter behavior.

## What I Changed and Why

I did not copy AI output blindly. I adapted and corrected it to match project requirements and C/POSIX behavior:

- Adjusted command flow so each operation integrates with district initialization and action logging.
- Refined permission logic to match role semantics (`manager` vs `inspector`) and required mode checks.
- Modified filter condition parsing and matching to support the actual struct fields and operators needed.
- Updated symlink handling to recreate `active_reports-<district>` reliably and remove stale links during cleanup.
- Improved error paths and return codes so failed operations terminate correctly and clearly.
- Kept naming, file layout, and behavior consistent with assignment expectations.

These changes were made to ensure correctness, consistency with the specification, and predictable runtime behavior.

## Critical Evaluation of AI Output

AI was useful for speed and brainstorming, but the output required careful verification:

- Some generated snippets were generic and did not fully match the assignment constraints.
- A few suggestions omitted important edge-case handling (e.g., file existence, mode validation, cleanup flow).
- Several responses proposed valid ideas but needed adaptation to the exact data structures and command contract used in this codebase.

Conclusion: AI improved development velocity, but correctness depended on manual review, testing, and iterative refinement.

## What I Learned

- AI is most effective for proposing structure and alternatives, not for final correctness.
- In systems programming, details matter: permissions, error handling, and file operations must be verified empirically.
- Comparing suggestions from **Cursor** and **Gemini** was useful to catch weak assumptions and improve confidence.
- The best workflow was: generate idea -> implement selectively -> test -> revise.
