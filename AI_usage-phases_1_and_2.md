# AI Usage Report - Phases 1 & 2

This document describes how AI tools were used during Phases 1 and 2 of the project.

## Phase 1

### Tools Used
- **Cursor** (AI coding assistant in IDE)
- **Gemini**

### Prompts I Gave
Examples of the kinds of prompts used:
- "Help me design a CLI structure for a C program with commands like add, list, view, remove, update threshold, and filter."
- "How should I safely create files/directories in C and enforce exact permission bits (0750, 0664, 0640, 0644)?"
- "Suggest a clean way to parse multiple filter conditions like `field:op:value` in C."

### What Was Generated by AI
AI-generated output included:
- Draft logic for command parsing and command dispatch.
- Example snippets for parsing conditions, validating access, and creating/updating symbolic links.

### What I Changed and Why
- Adjusted command flow to integrate with district initialization and action logging.
- Refined permission logic to match role semantics (`manager` vs `inspector`).
- Modified filter condition parsing and matching to support the actual struct fields and operators.

## Phase 2

### Tools Used
- **Gemini**

### Prompts I Gave
- "Write a C program called `monitor_reports` that creates a `.monitor_pid` file, waits for SIGINT to stop, and responds to SIGUSR1."
- "How to read a PID from a file and send a SIGUSR1 signal in C?"
- "Provide a structured README.md for the whole project. Include instructions regarding on how to run the code, what each component is doing"

### What Was Generated by AI
- Process creation and `exec` usage for safely calling `rm -rf`.
- Complete skeleton for `monitor_reports.c` including signal handling (`sigaction`).
- Code snippet to open `.monitor_pid`, read the process ID, and signal the monitor program using `kill()`.
- Error handling logic to record in the log file if the monitor could not be informed.
- A concise `README.md` describing project architecture, how to build using `make`, and examples showing how to interact with the programs.

### What I Changed and Why
- Integrated monitor notification directly into `add_report` within `operations.c`. Ensured that log messages clearly differentiate between success and failure in notifying the monitor.
- Set up proper signal blocking in the `monitor_reports.c` using `sigemptyset` to avoid signal races. Ensure that standard functions are used safely within signal handlers.
- Reviewed and applied AI's suggested `README.md` format, ensuring the usage commands match the implemented API exact structures for roles and commands.

### Critical Evaluation of AI Output
The AI was very helpful in quickly generating boilerplate code for signal handlers and `fork`/`exec` logic. However, I had to ensure the signal handling was minimal (e.g. using `write` instead of `printf` in handlers), and safely integrate the notification logic so that it only runs when `add_report` succeeds. Also, the prompt generated text for documentation needed verification against the exact implemented codebase to guarantee accurate usage examples.

## What I Learned
- POSIX signal handling requires specific care, such as only using async-signal-safe functions like `write` instead of `printf` in signal handlers.
- Safe usage of `fork` and `execl` system calls.
- Coordinating between multiple processes using signals and PID files.
- Utilizing AI not just for code, but to rapidly synthesize high-level architectural documentation (`README.md`).
